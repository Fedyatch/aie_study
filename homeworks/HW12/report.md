# HW12 – временные ряды: temporal split, baseline-модели и GRU-прогноз

## 1. Кратко: что сделано

- Использован датасет `S12-hw-dataset.csv` с ежечасным рядом `target`.
- Прогнозировался следующий час (`horizon = 1`).
- Разбиение: `train / validation / test` по времени (70/15/15), без перемешивания.
- Сравнивались 4 подхода:
  - `B1`: Naive-last (последнее значение)
  - `B2`: Moving-average (скользящее среднее за 24 часа)
  - `B3`: Ridge на лаговых, rolling и календарных признаках
  - `R1`: GRU на оконном представлении (window_size=48)
- Лучшей оказалась модель `B3` (Ridge): MAE = 6.14 на валидации, 7.23 на тесте.

## 3. Данные и постановка задачи

- Датасет: `S12-hw-dataset.csv`  
- Целевая колонка: `target`  
- Частота: hourly  
- Горизонт прогноза: 1  
- Размер: 4320 наблюдений  
- Диапазон дат: 2025-01-01 00:00:00 — 2025-06-29 23:00:00  
- Внешние признаки: нет (только `date`)  
- Комментарий: Ряд имеет выраженный восходящий тренд и суточную сезонность. Пропусков нет. Нестационарность делает задачу сложной для простых бейзлайнов.

## 4. Temporal split и признаки

### 4.1. Разбиение по времени
- Точное разбиение: `train` (3024), `val` (648), `test` (648) по хронологии  
- Доли: 70% / 15% / 15%  
- `random split` некорректен: нарушает временную зависимость, приводит к утечке будущего и завышенной оценке качества.

### 4.2. Признаки для baseline-моделей
- Лаги: `lag_1`, `lag_7`, `lag_14`, `lag_24`  
- Rolling: `rolling_mean_7`, `rolling_std_7`, `rolling_mean_24`  
- Календарные: `hour`, `dayofweek`, `is_weekend`, `hour_sin`, `hour_cos`  
- Пропуски: удалены после `.shift()` и `.rolling()`  
- Масштабирование: `StandardScaler`, обучен только на `train`

## 5. Модели и эксперименты (B1, B2, B3, R1)

- **B1 (naive-last)**: прогноз = последнее значение из train  
- **B2 (moving-average)**: прогноз = скользящее среднее за 24 часа  
- **B3 (ridge-lag-features)**: Ridge (alpha=1.0) на 16 инженерных признаках  
- **R1 (gru-forecast)**: 2-слойная GRU (hidden=64, window_size=48)  

Дополнительно:
- Основная метрика выбора: `val_MAE`
- Доп. метрики: `RMSE`, `MAPE`
- `window_size`: 48
- Batch size: 32
- Эпох: max 50 (ранняя остановка при 10 эпохах без улучшения)
- Optimizer: Adam
- LR: 1e-3
- Лучшая модель сохранялась по минимальному `val_loss`

## 7. Анализ

Наивный бейзлайн (B1) оказался самым слабым (MAE 20.34), так как игнорирует тренд и сезонность.  
Moving-average (B2) улучшил результат на 34% (MAE 13.40), но всё ещё не учитывает долгосрочные паттерны.  
Ridge (B3) показал лучшее качество благодаря явному моделированию временных зависимостей через признаки и регуляризации.  
GRU (R1) проиграла табличному подходу из-за ограниченного объёма данных (4320 точек) и высокой сложности модели (17.7k параметров).  
Тренд и суточная сезонность хорошо видны в ряде и отражаются в важности признаков (например, `lag_24`, `hour_sin`).  
Утечки данных возможны при случайном сплите, масштабировании на всём датасете или использовании `.shift(0)` вместо `.shift(1)`.  
Выбор по `validation`, а не `test`, критичен — тест должен использоваться один раз для финальной оценки.  
Наиболее показательные ошибки — в точках резкого изменения (например, после режимных сдвигов), где даже B3 даёт повышенную ошибку.

## 8. Итоговый вывод

- Базовый подход: **линейная регрессия с инженерными признаками** — оптимальна для малых/средних датасетов.
- Простой baseline на признаках достаточен, если данные имеют ясную структуру (тренд + сезонность) и объём < 10k.
- Переход к GRU/LSTM оправдан только при большом объёме данных (>50k), сложных нелинейностях или многошаговом прогнозе.
- Главное понимание: **temporal split обязателен**, любое нарушение хронологического порядка делает оценку некорректной.
