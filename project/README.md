# Итоговый проект по курсу «Инженерия Искусственного Интеллекта»

В этой папке находится итоговый мини-проект по курсу.  
Проект демонстрирует применение методов и инструментов инженерии ИИ: работу с данными, модели, пайплайны, сервис, эксперименты и воспроизводимость.

---

## 1. Паспорт проекта

- **Название проекта:** Сервис кредитного скоринга: предсказание вероятности дефолта заёмщика
- **Автор:** Чаусов Фёдор Валерьевич
- **Группа:** КМБО-11-24
- **Контакт:** tg - `@fedyatch`, email - `fedyatchausov@yandex.ru`
- **Ссылка на репозиторий:** [https://github.com/Fedyatch/aie_study.git]

- **Краткое описание:**  
  End‑to‑end сервис оценки кредитного риска на основе табличных данных. Задача бинарной классификации решается с помощью ансамблевых моделей (Logistic Regression, Random Forest, HistGradientBoosting), обученных на открытом датасете Give Me Some Credit (~150 000 записей). Реализован REST API на FastAPI, который по анкетным данным возвращает вероятность дефолта, категорию риска и время обработки. Архитектура включает модульный пайплайн предобработки, контейнеризацию через Docker, автоматические тесты и документированный процесс экспериментов.

## 4. Как запустить проект

### 4.1. Запуск обучения модели

Обучение проводится в Jupyter-ноутбуках:


```bash
cd project
uv run jupyter lab
```

Затем открыть и выполнить последовательно:
- `notebooks/01_eda_baseline.ipynb` — EDA и baseline
- `notebooks/02_model_tuning.ipynb` — обучение моделей, сохранение артефактов

### 4.2. Запуск сервиса (API/веб-интерфейс)

```bash
# Локальный запуск с горячей перезагрузкой
cd project
uv run uvicorn src.credit_scoring.api:app --reload
```

**Сервис доступен:**
- Основной адрес: `http://127.0.0.1:8000`
- Swagger UI (документация): `http://127.0.0.1:8000/docs`
- Health-check: `http://127.0.0.1:8000/health`

### 4.3. Запуск в Docker

```bash
# Сборка образа
cd project
docker build -t credit-scoring-api .

# Запуск контейнера
docker run -p 8000:8000 credit-scoring-api
```

### 4.4. Запуск в Docker
```bash
uv run pytest -q
```

## 7. Модели и эксперименты

### 7.1. Сравнение моделей

| Модель | ROC‑AUC | Accuracy | F1 | Время обучения |
|--------|---------|----------|----|----------------|
| Logistic Regression | 0.78 | 0.75 | 0.54 | 0.2 с |
| Random Forest | 0.80 | 0.76 | 0.56 | 1.5 с |
| **HistGradientBoosting** (финальная) | **0.80** | **0.76** | **0.56** | 0.8 с |

### 7.2. Обоснование выбора HistGradientBoosting

1. **Качество:** наилучший ROC‑AUC среди рассмотренных моделей.
2. **Производительность:** обучение < 1 с, инференс ~10 мс — отлично для real‑time API.
3. **Устойчивость:** встроенная обработка пропусков и выбросов, не требует отдельной импутации.
4. **Отсутствие внешних зависимостей:** входит в scikit-learn, в отличие от LightGBM.

Подробное обоснование — в `report.md`, раздел «Выбор финальной модели».

### 7.3. Воспроизводимость экспериментов

## 9. Ограничения и дальнейшая работа

### 9.1. Текущие ограничения

| Ограничение | Влияние | Обходное решение |
|-------------|---------|------------------|
| Дисбаланс классов (~14:1) | Снижение точности для дефолтов | SMOTE, `class_weight='balanced'`, оптимизация метрик |
| Отсутствие калибровки вероятностей | Вероятности могут быть смещены | `CalibratedClassifierCV` (Platt scaling) |
| Статическая модель | Деградация качества без переобучения | Мониторинг дрейфа данных, автоматическое переобучение |
| Нет аутентификации и rate limiting | Сервис открыт для неограниченного доступа | API‑ключи / JWT, лимитирование запросов |

### 9.2. Направления развития

1. **Интерпретируемость:** добавление SHAP‑анализа и endpoint `/explain`.
2. **Мониторинг:** Prometheus‑метрики, алерты при снижении качества.
3. **MLOps:** автоматический пайплайн переобучения (MLflow/Airflow), версионирование моделей.
4. **Масштабирование:** Kubernetes с горизонтальным масштабированием.
5. **Безопасность:** валидация входных данных, аудит логов.
