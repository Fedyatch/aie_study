- **Автор:** Чаусов Фёдор Валерьевич

**Краткое описание:**  
Проект реализует end‑to‑end сервис оценки кредитного риска. По анкетным данным заёмщика (10 числовых признаков) система возвращает вероятность дефолта и категорию риска. Решение упаковано в Docker‑контейнер, API построено на FastAPI, модель – HistGradientBoosting (sklearn). Проект демонстрирует полный цикл: от EDA и экспериментов до контейнеризованного сервиса с тестами и документацией.

## 2. Постановка задачи и контекст

**Предметная область:** кредитный скоринг – оценка вероятности того, что заёмщик допустит серьёзную просрочку (90+ дней) в течение двух лет.  
**Пользователь:** кредитный аналитик или автоматизированная система принятия решений.  
**Сценарий:** при поступлении новой заявки сервис мгновенно возвращает скоринговую оценку, позволяя снизить риски и ускорить обработку заявок.

**ML-постановка:** бинарная классификация (`SeriousDlqin2yrs`).  
- **Вход:** 10 числовых признаков (возраст, доход, долговая нагрузка, счётчики просрочек и др.).  
- **Выход:** вероятность дефолта и категория риска (`low`/`high`).  
- **Требования:** интерпретируемость (важность признаков), скорость инференса < 200 мс, устойчивость к пропускам и выбросам.

**Целевые метрики:**
- **ROC‑AUC** – основная метрика (устойчива к дисбалансу классов).
- **Accuracy, F1** – дополнительные метрики качества классификации.
- **latency_ms** – время обработки запроса в миллисекундах, замеряется в API и отображается в ответе.

## 3. Данные

**Источник:** [Give Me Some Credit (Kaggle)](https://www.kaggle.com/c/GiveMeSomeCredit).  
Датасет открытый, обезличенный, содержит ~150 000 записей о заёмщиках.

**Признаки (10 числовых):**
`RevolvingUtilizationOfUnsecuredLines`, `age`, `NumberOfTime30-59DaysPastDueNotWorse`, `DebtRatio`, `MonthlyIncome`, `NumberOfOpenCreditLinesAndLoans`, `NumberOfTimes90DaysLate`, `NumberRealEstateLoansOrLines`, `NumberOfTime60-89DaysPastDueNotWorse`, `NumberOfDependents`.

**Целевая переменная:** `SeriousDlqin2yrs` (0 – нет просрочки, 1 – есть).

**Ключевые наблюдения EDA (notebook `01_eda_baseline.ipynb`):**
- Сильный дисбаланс классов: ~93% недефолтов, ~7% дефолтов (≈14:1).
- Пропуски: `MonthlyIncome` (~20%), `NumberOfDependents` (~2.6%).
- Экстремальный перекос финансовых признаков (skewness > 95).
- Высокая корреляция между тремя счётчиками просрочек (|r| > 0.8).

**Предобработка (реализована в пайплайне):**
1. Медианная импутация пропусков.
2. Логарифмирование (`log1p`) для `RevolvingUtilizationOfUnsecuredLines`, `DebtRatio`, `MonthlyIncome`.
3. Агрегация трёх счётчиков просрочек в один признак `TotalPastDue`.
4. Масштабирование `RobustScaler`.

## 4. Модели и подходы

**Baseline:** `DummyClassifier` (most_frequent) показал ожидаемо нулевое качество для миноритарного класса (F1=0, ROC‑AUC=0.5), подтвердив наличие сигнала в данных.

**Рассмотренные модели (notebook `02_model_tuning.ipynb`):**
- **Logistic Regression** с подбором `C` (L2‑регуляризация) и `class_weight='balanced'`.
- **Random Forest** с подбором `n_estimators` и `max_depth`, `class_weight='balanced'`.
- **HistGradientBoosting** с подбором `learning_rate` и `max_depth`, `class_weight='balanced'`.

Все модели обучались на предобработанных данных с 5‑кратной стратифицированной кросс‑валидацией. Метрика оптимизации – ROC‑AUC.

---

## 5. Экспериментальный протокол и результаты

**Протокол:**
- Стратифицированное разбиение train/test (80/20), `random_state=42`.
- Гиперпараметры подбирались только на train через `GridSearchCV`.
- Финальная оценка – на отложенном тесте.

**Сравнение моделей на тесте:**

| Модель | ROC-AUC | Accuracy | F1 | Время инференса |
|--------|---------|----------|----|-----------------|
| Logistic Regression | 0.78 | 0.75 | 0.54 | < 5 мс |
| Random Forest | 0.80 | 0.76 | 0.56 | ~20 мс |
| **HistGradientBoosting** | **0.80** | **0.76** | **0.56** | ~10 мс |

Все модели значительно превосходят baseline. HistGradientBoosting показал наилучшее качество при сопоставимой скорости.

**Выбор финальной модели:** HistGradientBoosting.
- Немного лучший ROC‑AUC по сравнению с Random Forest и Logistic Regression.
- Быстрее Random Forest, не требует внешних библиотек (в отличие от LightGBM).
- Хорошо работает с пропусками и выбросами «из коробки».

Финальный пайплайн (предобработка + модель) сохранён в `artifacts/modeling/models/scorer_pipeline_v1.0.joblib`.

## 6. Архитектура решения и сервис

**Пайплайн обработки запроса:**
1. Валидация входного JSON через Pydantic-схему (`CreditApplication`).
2. Передача сырых признаков в сохранённый `sklearn.pipeline`.
3. Пайплайн выполняет импутацию, логарифмирование, агрегацию, масштабирование и предсказание.
4. Возвращается JSON с вероятностью дефолта, категорией риска и временем обработки.

**API (FastAPI, `src/credit_scoring/api.py`):**

| Endpoint | Метод | Описание |
|----------|-------|----------|
| `/health` | GET | Проверка работоспособности |
| `/predict` | POST | Скоринг заявки |

**Технологический стек:** Python 3.12, scikit-learn, FastAPI, Uvicorn, Pydantic, Docker.

**Запуск:**
```bash
# Локально
uv run uvicorn src.credit_scoring.api:app --reload

# Docker
docker build -t credit-scoring-api .
docker run -p 8000:8000 credit-scoring-api
```

## 7. Наблюдаемость, конфигурация и безопасность

### Логи и наблюдаемость
- В консоль логируются все запросы к `/predict`: выводится предсказанная вероятность дефолта и время обработки в миллисекундах (функция `print` в `api.py`).

- Реальные секреты (токены, пароли) в проекте отсутствуют. Файл `.env` добавлен в `.gitignore`, чтобы случайно не закоммитить локальные настройки.
- Шаблон `configs/.env.example` содержит только публичные параметры (порт).
- Данные проекта – открытый обезличенный датасет; никакая чувствительная информация не хранится.

## 8. Ограничения и дальнейшая работа

### Текущие ограничения
- Модель статична: нет механизма детекции дрейфа данных или автоматического переобучения.
- Сильный дисбаланс классов (~14:1) – точность (precision) на миноритарном классе остаётся невысокой.
- Калибровка вероятностей не выполнялась, поэтому значения `probability_default` могут быть смещены.
- Отсутствует мониторинг в production-стиле (Prometheus, алерты), а также авторизация и лимитирование запросов.

### Направления развития
- **Модель:**  
  - Калибровка вероятностей (`CalibratedClassifierCV`).  
  - Более широкий поиск гиперпараметров (Optuna, GridSearchCV).  
  - Эксперименты с альтернативными алгоритмами (LightGBM, XGBoost).  
- **Сервис:**  
  - Добавление Prometheus-метрик и алертов.  
  - Реализация кэширования и rate limiting.  
  - Версионирование моделей и A/B-тестирование.  
- **Данные:**  
  - Борьба с дисбалансом (SMOTE, взвешивание классов).  
  - Feature engineering: генерация дополнительных признаков на основе исходных.

## 9. Сценарий демонстрации на защите

1. **Запуск сервиса** – локально через `uv run uvicorn src.credit_scoring.api:app --reload` или в Docker.

5. **Ноутбуки** – показываю `01_eda_baseline.ipynb` (графики распределений, baseline) и `02_model_tuning.ipynb` (таблицу сравнения моделей, ROC-кривые, важность признаков).
6. **Тесты** – запускаю `uv run pytest -q`, показываю успешное прохождение двух тестов.
7. **Архитектура** – кратко объясняю путь запроса: Pydantic-валидация → препроцессинг внутри пайплайна → `HistGradientBoosting` → ответ.

Команды и примеры запросов полностью соответствуют инструкциям из `project/README.md`.
